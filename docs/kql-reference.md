# KQL Reference

This document provides a practical Kusto Query Language (KQL) reference for analyzing Defender for Cloud SQL test alerts in Microsoft Sentinel, Microsoft Defender XDR, and related Azure logging sources.

## Introduction to KQL for Defender alerts

KQL helps you:

- Search for SQL-related alerts generated by Defender for Cloud
- Correlate alerts with entities, evidence, and diagnostic logs
- Build summaries, timelines, and validation dashboards
- Confirm whether a simulation produced the expected detection

Use this guide together with the repository queries under `kql\sentinel\` and `kql\xdr\`.

## Table reference

### Core tables

| Table | Platform | Use case |
|---|---|---|
| `SecurityAlert` | Microsoft Sentinel / Log Analytics | Primary table for Defender and Sentinel alert records |
| `AlertInfo` | Microsoft Defender XDR | High-level alert metadata |
| `AlertEvidence` | Microsoft Defender XDR | Entity and evidence records tied to alerts |
| `AzureDiagnostics` | Azure Monitor / Log Analytics | SQL audit and diagnostic records |

### Table purpose details

#### `SecurityAlert`

Use for:

- Alert title, severity, product source, and timestamps
- Extended JSON properties
- Entity extraction and Sentinel reporting

Common columns include:

- `TimeGenerated`
- `AlertName`
- `AlertSeverity`
- `CompromisedEntity`
- `ProductName`
- `ProviderName`
- `ExtendedProperties`
- `Entities`
- `ResourceId`

#### `AlertInfo`

Use for:

- XDR alert metadata
- Classification, determination, service source, and severity
- Alert timeline analysis

Common columns include:

- `Timestamp`
- `Title`
- `Severity`
- `ServiceSource`
- `Category`
- `DetectionSource`
- `AttackTechniques`
- `AlertId`

#### `AlertEvidence`

Use for:

- Accounts, devices, IP addresses, URLs, and process evidence tied to alerts
- Joining back to `AlertInfo` with `AlertId`

Common columns include:

- `Timestamp`
- `AlertId`
- `EntityType`
- `DeviceName`
- `AccountName`
- `RemoteIP`
- `EvidenceRole`

#### `AzureDiagnostics`

Use for:

- SQL auditing validation
- Login activity review
- Supplementing alert evidence with raw events

Common columns vary by diagnostic category, but often include:

- `TimeGenerated`
- `ResourceType`
- `Category`
- `OperationName`
- `LogicalServerName_s`
- `database_name_s`
- `client_ip_s`
- `statement_s`

## Schema details for SQL alerts

### Common fields

The most useful fields across Sentinel and XDR are:

| Concept | Sentinel field | XDR field |
|---|---|---|
| Timestamp | `TimeGenerated` | `Timestamp` |
| Alert name | `AlertName` | `Title` |
| Severity | `AlertSeverity` | `Severity` |
| Alert identifier | `SystemAlertId` or record ID | `AlertId` |
| Source product | `ProductName` | `ServiceSource` |
| Resource/entity | `CompromisedEntity` | Device, account, or resource fields in evidence |

### ExtendedProperties parsing

Sentinel alerts commonly store additional context as JSON in `ExtendedProperties`.

Example pattern:

```kusto
SecurityAlert
| where TimeGenerated > ago(24h)
| where ProductName has "Defender"
| extend ParsedProperties = parse_json(ExtendedProperties)
| project TimeGenerated, AlertName, AlertSeverity,
          ResourceId,
          ExtendedProperties,
          ParsedProperties
```

To safely extract a field from the JSON:

```kusto
SecurityAlert
| where TimeGenerated > ago(24h)
| extend ParsedProperties = parse_json(ExtendedProperties)
| extend SimulationName = tostring(ParsedProperties.SimulationName)
| extend SqlServerName = tostring(ParsedProperties.ServerName)
| project TimeGenerated, AlertName, AlertSeverity, SimulationName, SqlServerName
```

### Entity extraction

The `Entities` column may also contain JSON arrays.

```kusto
SecurityAlert
| where TimeGenerated > ago(24h)
| extend ParsedEntities = parse_json(Entities)
| mv-expand ParsedEntities
| project TimeGenerated, AlertName, Entity = ParsedEntities
```

For XDR evidence correlation:

```kusto
AlertInfo
| where Timestamp > ago(24h)
| where Title has "SQL"
| join kind=leftouter (
    AlertEvidence
    | where Timestamp > ago(24h)
) on AlertId
| project Timestamp, Title, Severity, EntityType, DeviceName, AccountName, RemoteIP
```

## Query templates

### Time-based filtering patterns

#### Last hour

```kusto
SecurityAlert
| where TimeGenerated > ago(1h)
| where ProductName has "Defender"
```

#### Custom time window

```kusto
let startTime = datetime(2025-01-01 10:00:00);
let endTime = datetime(2025-01-01 12:00:00);
SecurityAlert
| where TimeGenerated between (startTime .. endTime)
| where AlertName has "SQL"
```

### Aggregation patterns

#### Alerts by type

```kusto
SecurityAlert
| where TimeGenerated > ago(24h)
| summarize AlertCount = count() by AlertName, AlertSeverity
| order by AlertCount desc
```

#### Alerts by hour

```kusto
SecurityAlert
| where TimeGenerated > ago(24h)
| summarize AlertCount = count() by bin(TimeGenerated, 1h), AlertName
| order by TimeGenerated asc
```

### Joining tables

#### Join XDR alert metadata to evidence

```kusto
AlertInfo
| where Timestamp > ago(24h)
| where Title has "SQL"
| join kind=leftouter (
    AlertEvidence
    | where Timestamp > ago(24h)
) on AlertId
| project Timestamp, Title, Severity, ServiceSource, EntityType, DeviceName, AccountName, RemoteIP
```

#### Join Sentinel alerts to SQL diagnostics by resource and time window

```kusto
let sqlAlerts = SecurityAlert
| where TimeGenerated > ago(24h)
| where AlertName has "SQL"
| project AlertTime = TimeGenerated, AlertName, ResourceId;
AzureDiagnostics
| where TimeGenerated > ago(24h)
| where ResourceType has "DATABASES" or ResourceType has "SERVERS"
| join kind=inner sqlAlerts on ResourceId
| where abs(datetime_diff('minute', TimeGenerated, AlertTime)) <= 30
| project AlertTime, AlertName, DiagnosticTime = TimeGenerated, OperationName, Category, ResourceId
```

## Alert type reference

Use this table as a quick mapping aid when reviewing test results.

| Alert name | Alert type code | Severity | MITRE tactic | Description |
|---|---|---|---|---|
| Brute Force | `SQL.VM_BruteForce` | Medium | Credential Access | Multiple failed SQL authentication attempts against a SQL VM |
| SQL Injection | `SQL.VM_PotentialSqlInjection` | High | Initial Access / Execution | Malicious SQL statement or payload pattern detected |
| Suspicious App | `SQL.DB_HarmfulApplication` | Medium | Execution | Access by a suspicious or harmful application |
| Principal Anomaly | `SQL.DB_PrincipalAnomaly` | Medium | Persistence / Credential Access | Login from a principal not commonly observed |
| Shell Anomaly | `SQL.VM_ShellExternalSourceAnomaly` | High | Discovery / Execution | External-source shell behavior observed on SQL VM |
| Obfuscation | `SQL.VM_PotentialSqlInjection` | High | Defense Evasion | Obfuscated SQL payload intended to evade simple pattern detection |

## Customization tips

### Modify time ranges

Replace `ago(24h)` with other windows such as:

- `ago(15m)` for near-real-time testing
- `ago(2h)` for a single validation session
- `ago(7d)` for trend review

### Add filters

Common filter ideas:

```kusto
| where AlertSeverity =~ "High"
| where CompromisedEntity has "sqltest"
| where ResourceId has "rg-defender-sql-test"
| where Title has_any ("SQL", "Injection", "Brute")
```

### Create workbooks from queries

In Sentinel or Azure Monitor:

1. Start with a validated query.
2. Add visual-friendly projections and summaries.
3. Pin the results to a workbook.
4. Use parameters for time range, resource group, or alert severity.
5. Save a workbook version dedicated to SQL simulation validation.

## Example validation workflow

1. Run a simulation.
2. Query `SecurityAlert` for the expected alert name.
3. Query `AlertInfo` and `AlertEvidence` to confirm evidence enrichment.
4. Query `AzureDiagnostics` for related audit or connection activity.
5. Save the final query set for repeatable testing.
